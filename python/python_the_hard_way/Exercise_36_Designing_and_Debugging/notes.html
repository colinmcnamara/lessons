designing and debugging

Rules for if-Statements

  1. Every if-statement must have an else

  2. if this else should never be run because it doesn’t make sense

    1. then you must use a die function in the else that prints out an error message and dies, (just like the dead function in the bear room excersize)

  3. Never nest if-statements more than two deep and always try to do them one deep.

    1. This means if you put an if and an if then you should be looking to move that second if to another function

  4. Treat if-statemnts like paragraphs, where each if elif else grouping is like a set of sentances.

    1. put blank lines before and after

  5. Your bolean tests should be simple.

    1. if they are complex. move the calculations to variables earlier in your function.

    2. use a good name for your variable

Rules for loops

  1. use a while-loop only to loop forever.

    1. that pretty much means never

  2. use a for-loop for all other kinds of lopping.

    1. especially if there is a fixed or limited number of things to do

Debuging tips

  1. Do not use a debuger

    1. you get to much information

  2. use print statements throughout your code

  3. Make sure little parts of your code work

    1. do not write massive stuff

    2. write tests throughout

    3. code a bit, test a bit, etc etc
